# Governing Agents with Azure Container Apps Sandboxes

> **Packages:** `agent-sandbox`, `agent-os-kernel`, `azure-sandbox`, `azure-mgmt-sandbox`
> **Audience:** Platform engineers running governed AI agents on Azure
> **Prerequisites:** Azure subscription with Container Apps sandbox preview enabled, Python 3.11+

---

## Why pair AGT with Azure sandboxes

Agent Governance Toolkit (AGT) handles **policy** — what an agent is allowed to
do. The Azure Container Apps sandbox preview handles **isolation** — where the
agent's code actually runs. Each one alone leaves a gap:

- AGT policy alone cannot stop a runaway `subprocess` call on the host that
  loaded the agent.
- An Azure sandbox alone has no opinion on which tool calls are safe, which
  delegations are valid, or which audit envelope an output belongs in.

`AgentGovernance Toolkit` plugs these together through the
[`SandboxProvider` ABC](DOCKER-SANDBOX-ISOLATION-DESIGN.md#generic-sandboxprovider-interface).
The Azure backend, [`ACASandboxProvider`](AZURE-SANDBOX-ISOLATION-DESIGN.md),
maps each governed session onto an Azure Container Apps sandbox inside a
sandbox group while running the same `PolicyEvaluator` gate as every other
backend.

The picture:

```
┌─────────────────────────────────────────────────────────────────┐
│  Your application                                                │
│    PolicyDocument.from_yaml("research_policy.yaml")              │
│                            │                                     │
│                            ▼                                     │
│    ACASandboxProvider.create_session_async(agent_id, policy)   │
└──────────────┬───────────────────────────────┬──────────────────┘
               │ azure-sandbox (data plane)    │ azure-mgmt-sandbox
               ▼                               ▼ (ARM)
   ┌──────────────────────┐      ┌────────────────────────────┐
   │ SandboxClient        │      │ SandboxGroupManagement-    │
   │  • create_sandbox    │      │  Client                    │
   │  • set_egress_policy │      │  • create_group            │
   │  • exec              │      │  • get_group / delete_group│
   │  • delete_sandbox    │      └────────────────────────────┘
   └──────────┬───────────┘
              ▼
   management.containerapps.azure.com
   (per-sandbox container with managed identity + egress proxy)
```

The two SDKs come from the
[`azure-sandbox`](https://pypi.org/project/azure-sandbox/)
and
[`azure-mgmt-sandbox`](https://pypi.org/project/azure-mgmt-sandbox/)
packages. AGT does not vendor them — it only depends on their public
surface, so any update Microsoft ships to those clients flows through unchanged.

### `ACASandboxProvider` API at a glance

The full surface that
[`agent_sandbox.ACASandboxProvider`](../../agent-governance-python/agent-sandbox/src/agent_sandbox/aca_sandbox_provider/aca_sandbox_provider.py)
adds on top of the `SandboxProvider` ABC is small. Everything below has
an `_async` twin (auto-generated by the ABC, currently
`asyncio.to_thread` wrappers) — async forms are what every example in
this article uses.

| Method | What it does | When you call it |
|--------|--------------|------------------|
| `__init__(resource_group, sandbox_group, *, subscription_id=None, credential=None, disk="ubuntu", ensure_group_location=None)` | Constructs the provider, lazily imports `azure-sandbox` / `azure-mgmt-sandbox`, builds the `SandboxClient` and `SandboxGroupManagementClient`. **Does not call Azure.** Pass `disk=<image_id>` to boot from a custom sandbox image; pass `ensure_group_location="<region>"` to auto-create the sandbox group on first use. | Once per process — the provider is reusable across many sessions. |
| `is_available() -> bool` | Returns `True` iff `azure-sandbox` was importable. Cheap; no network. | Health checks, feature flags, optional-dependency guards before any other call. |
| `create_session(agent_id, *, policy=None, config=None) -> SessionHandle` | Provisions a sandbox VM via `SandboxClient.create_sandbox`, applies the policy-derived egress rules via `set_egress_policy`, and binds a `PolicyEvaluator` for that `(agent_id, session_id)`. Returns a `SessionHandle` whose `session_id` is the Azure sandbox id. | Once per agent task. The async form `create_session_async` is what Step 4 / Step 5 use. |
| `execute_code(agent_id, session_id, code, *, context=None) -> ExecutionHandle` | Runs the `PolicyEvaluator` host-side gate, then base64-encodes the source and pipes it into `python3` inside the sandbox via `SandboxClient.exec`. Wraps `exitCode` / `stdout` / `stderr` in a `SandboxResult` and applies the `timeout_seconds` kill behavior. Raises `PermissionError` on policy deny *before* any Azure traffic. | Once per agent step. `context=` is merged into the policy evaluation context (see [Step 5.2](#52--the-agent)). |
| `destroy_session(agent_id, session_id) -> None` | Calls `SandboxClient.delete_sandbox` and removes the in-memory bookkeeping. **Idempotent** — safe from `finally:`, signal handlers, and test teardown even when the bookkeeping is already gone. | Always, in a `finally` (or via `__aexit__`) so sandbox VMs are not left billing. |
| `get_session_status(agent_id, session_id) -> SessionStatus` | Returns the in-memory status (`READY` / `CLOSED` / unknown). Does not query Azure. | Reconciliation between your bookkeeping and the provider's; rarely needed in normal flows. |
| `close() -> None` | Best-effort cleanup of every still-open session this provider created, then closes the underlying `SandboxClient` / `SandboxGroupManagementClient` HTTP connections. | At process shutdown, or as a safety net in long-lived services. |
| `provider._data_client` (escape hatch) | The raw `azure.sandbox.SandboxClient`. Use this for anything not covered by the ABC: `create_disk_image`, `commit_sandbox`, `create_snapshot`, `get_egress_decisions`, port forwards, lifecycle policies. | When you need an Azure-specific feature — see [Step 6](#step-6--drop-down-to-the-sdks-when-you-need-to). |
| `provider._mgmt_client` (escape hatch) | The raw `azure.mgmt.sandbox.SandboxGroupManagementClient`. ARM-side: groups, role assignments, tags. | Bootstrapping or platform-team automation that is not handled by `ensure_group_location` — see [Step 3](#step-3--bootstrap-the-resource-group-and-sandbox-group). |

A few rules of thumb:

- **Use `*_async` forms.** Every example below does. The async wrappers
  cost one thread-pool hop per call, which is dominated by the
  ~50–500 ms data-plane round-trip anyway, and they keep your event
  loop responsive when you run multiple sessions concurrently.
- **One `policy=` per session, not per call.** The policy is bound at
  `create_session_async` time and reused on every `execute_code_async`
  call against that handle. To change policy, destroy the session and
  create a new one with the new `PolicyDocument`.
- **Stay above `_data_client` whenever you can.** Code that reaches into
  the underscore-prefixed attributes is portable to Azure only — the
  Docker and Hyperlight backends do not have them. The ABC surface
  (`create_session_async` / `execute_code_async` / `destroy_session_async`)
  is what works across every backend AGT ships.

## What the integration covers

| AGT concern | Where it is enforced | Mechanism |
|-------------|----------------------|-----------|
| **Policy decisions** (allow/deny per call) | Host process | `PolicyEvaluator` runs *before* every `exec` call |
| **CPU / memory caps** | Azure sandbox VM | `defaults.max_cpu` / `defaults.max_memory_mb` mapped to sandbox `resources.cpu` / `resources.memory` |
| **Network allowlist** | Azure egress proxy | `network_allowlist` mapped to `egressPolicy.hostRules` with `defaultAction=Deny` |
| **Tool allowlist** | Host process | Checked by `PolicyEvaluator` before exec — the sandbox VM cannot call back into host tools |
| **Lifecycle / suspend** | Azure | `auto_suspend_seconds` parameter on `create_sandbox` |
| **Audit** | AGT trust + Azure | AGT receipts capture the policy decision; Azure captures egress decisions via `get_egress_decisions` |

Anything outside the `SandboxProvider` ABC — snapshots, `commit_sandbox`,
ports, telemetry — remains reachable through `provider._data_client` for
power users. The contract this article walks through stays inside the ABC.

## Step 1 — Install

```bash
pip install agent-sandbox agent-os-kernel
pip install azure-sandbox azure-mgmt-sandbox azure-identity
```

Why each package:

- `agent-sandbox` — the `ACASandboxProvider` adapter and the
  `SandboxProvider` ABC.
- `agent-os-kernel` — the `PolicyDocument` + `PolicyEvaluator` used by the
  per-call gate.
- `azure-sandbox` / `azure-mgmt-sandbox` — Microsoft's first-party SDKs
  (data plane and ARM control plane) for the sandbox preview.
- `azure-identity` — supplies `DefaultAzureCredential`; both Azure clients
  default to it.

Verify the wiring:

```python
from agent_sandbox import ACASandboxProvider
provider = ACASandboxProvider(resource_group="agents-rg", sandbox_group="agents")
assert provider.is_available(), "azure-sandbox SDK or credentials missing"
```

`is_available()` returns `False` (and refuses to create sessions) if
`azure-sandbox` could not be imported. The constructor itself does **not**
call Azure — it is safe to instantiate the provider before the user has
run `az login`.

## Step 2 — Author a policy

`ACASandboxProvider` reads four things off the policy object you pass
to `create_session`:

- **`policy.rules`** — `PolicyDocument` rules. Evaluated host-side by
  `PolicyEvaluator` on every `execute_code` call. A `deny` raises
  `PermissionError` *before* any Azure traffic.
- **`policy.network_allowlist`** — list of host patterns the sandbox
  is allowed to reach. Combined with `defaults.network_default` to form
  the sandbox egress policy.
- **`policy.defaults.network_default`** — `"deny"` (the default,
  fail-closed) or `"allow"`. See ["Egress is fail-closed by default"](#egress-is-fail-closed-by-default)
  below.
- **`policy.tool_allowlist`** and **`policy.defaults.max_cpu` /
  `policy.defaults.max_memory_mb` / `policy.defaults.timeout_seconds`**
  — used to constrain the host-side tool surface, sandbox VM size, and
  per-step wall-clock cap.

> **Schema note.** As of AGT 3.3, `network_allowlist`, `tool_allowlist`,
> `max_cpu`, `max_memory_mb`, `timeout_seconds`, and `network_default`
> are **native fields on `PolicyDocument`** — no `SimpleNamespace`
> wrapper is required. Older code that wrapped policies via
> `types.SimpleNamespace(...)` still works (the provider keeps reading
> via `getattr`), but new code should declare the fields directly in
> YAML. See [`examples/quickstart/azure_sandbox_step5_test.py`](../../examples/quickstart/azure_sandbox_step5_test.py)
> for a runnable end-to-end example.

A typical policy looks like this — every field below is part of the
native `PolicyDocument` schema, so a single `PolicyDocument.from_yaml(...)`
call returns a fully-formed policy object ready to pass to
`create_session(..., policy=document)`.

```yaml
# research_policy.yaml — all fields below are native PolicyDocument keys.

name: research-agent
version: "1"

defaults:
  action: allow              # PolicyDocument rule-engine default action
  max_cpu: 0.5               # → sandbox resources.cpu = "500m"
  max_memory_mb: 1024        # → sandbox resources.memory = "1024Mi"
  timeout_seconds: 60        # → execute_code wall-clock kill threshold
  network_default: deny      # fail-closed; anything not in allowlist is denied

network_allowlist:
  - "*.github.com"
  - "pypi.org"
  - "api.openai.com"

tool_allowlist:
  - web_search
  - read_doc

rules:
  # PolicyEvaluator runs each rule's condition against the eval context.
  # The provider passes context = {"action": "execute", "code": <code>, ...}
  # so a CONTAINS check on the `code` field is how you gate snippets.
  - name: deny-shell-out-subprocess
    condition:
      field: code
      operator: contains
      value: "subprocess"
    action: deny
    priority: 100
    message: "shell-out blocked by research-agent policy"

  - name: deny-shell-out-os-system
    condition:
      field: code
      operator: contains
      value: "os.system("
    action: deny
    priority: 100
    message: "shell-out blocked by research-agent policy"
```

To use it, load the YAML and pass the document straight through:

```python
from agent_os.policies import PolicyDocument

policy = PolicyDocument.from_yaml("research_policy.yaml")
# `policy` is now ready to pass to provider.create_session(..., policy=policy).
```

### Egress is fail-closed by default

When you pass a `PolicyDocument` to `create_session`, the provider
always writes an egress policy to the sandbox — there is no path where
a policy-governed session inherits Azure's default-allow behavior by
accident. The behavior is controlled by `defaults.network_default`:

| `network_default` | `network_allowlist` | Resulting sandbox egress |
|-------------------|---------------------|--------------------------|
| `deny` *(default)* | `[]` | **Deny all** — sandbox has no outbound network |
| `deny` *(default)* | `["pypi.org", ...]` | Allow only the listed hosts; deny everything else |
| `allow` | *(ignored)* | Allow all — no egress API call is made |

The `"deny"` default is the recommended posture for production agents.
The `"allow"` mode exists for trusted dev/research workloads where the
rule engine is the only intended gate; using it in production is a
policy decision that should be reviewed.

If you do not pass a policy at all (`provider.create_session(agent_id)`
with no `policy=`), no egress policy is written — the sandbox keeps
Azure's default-allow behavior. This matches the behavior of a raw
`azure-sandbox` session and is intentional so the provider stays usable
for non-governed workloads, but it means **every governed session must
pass a policy**.

### How `network_allowlist` becomes an Azure egress policy

The provider does this translation for you inside `create_session`. The
relevant code lives in
[`agent_sandbox.aca_sandbox_provider`](../../agent-governance-python/agent-sandbox/src/agent_sandbox/aca_sandbox_provider/aca_sandbox_provider.py)
and is intentionally small:

```python
# agent_sandbox/aca_sandbox_provider/aca_sandbox_provider.py (excerpt)

def _network_allowlist(policy):
    """Return the policy's network allowlist as a list of host patterns."""
    out = []
    for entry in getattr(policy, "network_allowlist", None) or []:
        if isinstance(entry, str):
            out.append(entry)
        else:
            host = getattr(entry, "host", None) or getattr(entry, "pattern", None)
            if host:
                out.append(host)
    return out

def _network_default(policy):
    """Fail-closed: returns 'deny' unless the policy explicitly opts out."""
    defaults = getattr(policy, "defaults", None)
    value = getattr(defaults, "network_default", None) if defaults else None
    if isinstance(value, str) and value.lower() in ("allow", "deny"):
        return value.lower()
    return "deny"

def _apply_egress_policy(self, sandbox_id, hosts, network_default="deny"):
    if network_default == "allow":
        return  # explicit opt-out — leave Azure's default-allow behavior
    # Always POST a deny policy when fail-closed, even if `hosts` is empty.
    policy = {
        "defaultAction": "Deny",
        "hostRules": [{"pattern": h, "action": "Allow"} for h in hosts],
    }
    self._data_client.set_egress_policy(
        sandbox_id, self._sandbox_group, policy,
        resource_group=self._resource_group,
    )
```

The call sequence inside `create_session` is:

1. `create_sandbox` provisions the container. By default, sandboxes have an
   *implicit* `defaultAction = "Allow"` egress policy — so until step 3
   the new sandbox can reach the open internet.
2. `_network_allowlist(policy)` and `_network_default(policy)` extract
   the host list and the default-action from the policy document. Both
   the YAML list-of-strings form and the object form (`{host: "..."}`
   or `{pattern: "..."}`) are accepted for the allowlist.
3. `_apply_egress_policy(sandbox_id, hosts, network_default)` POSTs an
   egress policy to `…/sandboxes/{id}/egresspolicy` via
   [`SandboxClient.set_egress_policy`](https://pypi.org/project/azure-sandbox/).
   The body is a single deterministic shape:

   ```json
   {
     "defaultAction": "Deny",
     "hostRules": [
       {"pattern": "api.openai.com",  "action": "Allow"},
       {"pattern": "api.arxiv.org",   "action": "Allow"},
       {"pattern": "*.github.com",    "action": "Allow"},
       {"pattern": "pypi.org",        "action": "Allow"}
     ]
   }
   ```

For the policy in the YAML above, the resulting egress policy is exactly:

| Policy YAML entry | Generated Azure host rule |
|-------------------|---------------------------|
| `"*.github.com"` | `{"pattern": "*.github.com", "action": "Allow"}` |
| `"pypi.org"`     | `{"pattern": "pypi.org",     "action": "Allow"}` |
| `"api.openai.com"` | `{"pattern": "api.openai.com", "action": "Allow"}` |
| *(everything else)* | covered by `defaultAction: "Deny"` |

A few intentional properties of this mapping:

- **Wildcards are passed through verbatim.** `*.github.com` becomes the
  Azure proxy's wildcard host pattern; AGT does not expand or rewrite it.
- **Order does not matter.** Each entry becomes an independent `Allow`
  rule. The `defaultAction: "Deny"` catches everything not listed.
- **An empty `network_allowlist` with `network_default: deny` is a true
  deny-all sandbox.** The provider sends
  `{"defaultAction": "Deny", "hostRules": []}`, which the Azure egress
  proxy applies fail-closed. No sentinel host trick is needed.
- **Calling `create_session` without a policy** writes no egress policy
  at all, so the sandbox keeps Azure's default-allow behavior. Every
  governed session must pass a policy.
- **Failures are logged, not raised.** A 5xx from the egress endpoint is
  warned and the session continues with whatever policy was previously
  applied. Production deployments should monitor that warning and treat
  any occurrence as a security incident.

### Two important properties of the mapping

1. **Sandboxes governed by a policy are always fail-closed.** Every
   call to `create_session(..., policy=p)` writes an egress policy.
   With the schema default (`defaults.network_default: deny`), forgetting
   a host from `network_allowlist` does not silently grant access — the
   request is denied at the Azure proxy. To get a default-allow sandbox
   you must set `network_default: allow` explicitly in the policy.
2. **`tool_allowlist` is host-side, not VM-side.** The Azure sandbox VM has
   no way to call back into your Python process to invoke a tool, so the
   allowlist is enforced by the in-process `PolicyEvaluator` *before* the
   call is base64-piped into the sandbox. Inside-the-VM tool brokering
   would require running your own proxy in the sandbox and adding it to
   `network_allowlist`.

## Step 3 — Bootstrap the resource group and sandbox group

The **resource group must already exist** — `ACASandboxProvider` does not
create or manage resource groups. Create one out-of-band first:

```bash
az group create --name agents-rg --location westus2
```

The **sandbox group** (the sub-resource inside the RG) *can* be
auto-created by the provider on demand:

```python
from agent_sandbox import ACASandboxProvider

provider = ACASandboxProvider(
    resource_group="agents-rg",       # MUST already exist
    sandbox_group="agents",
    ensure_group_location="westus2",  # creates sandbox group if missing
)
```

Under the hood this calls `SandboxGroupManagementClient.create_group` only
when `get_group` returns 404, so the call is idempotent. If the enclosing
resource group is missing, the create call surfaces Azure's
`ResourceGroupNotFound` 404 wrapped in a `RuntimeError` from
`create_session`.

For tighter control (custom managed identity, tags, separate
provisioning pipeline), do it yourself once:

```python
from azure.mgmt.sandbox import SandboxGroupManagementClient

with SandboxGroupManagementClient(resource_group="agents-rg") as mgmt:
    mgmt.create_group(
        "agents",
        location="westus2",
        identity={"type": "SystemAssigned"},
        tags={"workload": "ai-agents", "owner": "platform-team"},
    )
```

Then build the provider without `ensure_group_location` so it does not try
to PATCH the group on every restart.

## Step 4 — Run a governed session end to end

All examples in this article use the async API. The provider's async
methods are supplied by the `SandboxProvider` ABC and currently delegate
to the sync implementations via `asyncio.to_thread`, so each call costs
one thread-pool hop. When `azure-sandbox` exposes a native async client,
the provider can override the async methods directly without changing
caller code.

```python
import asyncio

from agent_sandbox import ACASandboxProvider
from agent_os.policies import PolicyDocument

async def main() -> None:
    # All sandbox fields (network_allowlist, tool_allowlist, max_cpu,
    # max_memory_mb, timeout_seconds, network_default) are native to
    # PolicyDocument — no wrapper required.
    policy = PolicyDocument.from_yaml("research_policy.yaml")

    provider = ACASandboxProvider(
        resource_group="agents-rg",
        sandbox_group="agents",
        disk="python-3.13",   # public Python sandbox image; default "ubuntu" has no python3
    )
    handle = await provider.create_session_async(
        "research-agent-1", policy=policy,
    )

    try:
        # 1. Allowed by policy AND inside the egress allowlist → succeeds.
        ok = await provider.execute_code_async(
            "research-agent-1",
            handle.session_id,
            "from urllib.request import urlopen\n"
            "with urlopen('https://pypi.org', timeout=10) as r: print(r.status)\n",
        )
        print("allowed:", ok.result.stdout.strip())  # → 200

        # 2. Blocked by the policy's network_allowlist, enforced at the Azure
        #    egress proxy. The host-side rule engine does not inspect URLs
        #    inside arbitrary Python source — that responsibility is delegated
        #    to the egress proxy, which the provider configured to fail-closed
        #    from `network_allowlist` during create_session_async. example.com
        #    is not in research_policy.yaml, so the request returns 403 from
        #    the proxy.
        blocked_egress = await provider.execute_code_async(
            "research-agent-1",
            handle.session_id,
            "from urllib.request import urlopen\n"
            "try:\n"
            "    with urlopen('https://example.com', timeout=10) as r: print('status', r.status)\n"
            "except Exception as exc: print('egress-blocked', type(exc).__name__, exc)\n",
        )
        print("blocked by network_allowlist:", blocked_egress.result.stdout.strip())

        # 3. Denied by an AGT policy *rule* before the call leaves the host —
        #    rule-engine deny, raised as PermissionError, no Azure traffic.
        try:
            await provider.execute_code_async(
                "research-agent-1",
                handle.session_id,
                "import subprocess; subprocess.check_output(['ls', '/'])",
            )
        except PermissionError as exc:
            print("denied by policy rule:", exc)
    finally:
        await provider.destroy_session_async(
            "research-agent-1", handle.session_id,
        )

asyncio.run(main())
```

What is happening on each `execute_code_async` call:

1. **Rule-engine policy gate (host-side).** `PolicyEvaluator` evaluates
   `{"agent_id": "research-agent-1", "action": "execute", "code": <src>}`
   against the `rules:` section of the policy. A deny short-circuits with
   `PermissionError` — no Azure traffic is generated. This is what catches
   case 3 above (`subprocess`).
2. **Transport.** The source is base64-encoded and piped into `python3`
   inside the sandbox via `client.exec(sandbox_id, group, "echo <b64> |
   base64 -d | python3")`. Encoding eliminates shell quoting hazards even
   for multi-line, quote-heavy, or unicode source.
3. **Sandbox execution + network policy gate (Azure-side).** The sandbox
   container runs the code under its policy-derived CPU and memory caps.
   Outbound HTTP goes through the Azure egress proxy, which applies the
   host rules derived from the policy's `network_allowlist` during
   `create_session`. This is what catches case 2 above (`example.com`) —
   it is still a policy denial, just enforced at the proxy rather than at
   the rule engine, because `PolicyEvaluator` does not parse URLs out of
   arbitrary Python source.
4. **Result wrapping.** `exitCode`, `stdout`, and `stderr` come back in the
   data-plane response and are wrapped in a `SandboxResult` with
   stdout / stderr truncated to 10 KB. If wall-clock time exceeded
   `cfg.timeout_seconds`, `SandboxResult.killed` is set so callers can
   detect runaway executions even though the data-plane API does not
   surface a kill signal.

`destroy_session_async` always calls `client.delete_sandbox`, even when
the in-memory bookkeeping is missing, so it is safe to call from
`finally:` blocks, async context managers, signal handlers, or test
teardown.

## Step 5 — A complete governed research agent

This is the kind of code a customer would actually deploy: a research
agent that reads tickets from an internal queue, asks an LLM to plan
analysis steps, runs each step as Python inside an Azure sandbox under a
strict policy, and returns a structured report with an audit receipt.

The example pulls together everything from the previous steps:

- A real `PolicyDocument` with `network_allowlist`, `tool_allowlist`, and
  rule-based `deny` decisions.
- LLM-generated code that is **never trusted** — every snippet goes
  through the AGT policy gate, then runs inside an Azure sandbox under
  CPU/memory caps with a fail-closed egress policy.
- Per-step receipts so the platform team can prove what the agent did,
  what the policy decided, and what the Azure egress proxy decided.
- Defensive handling for the realistic failure modes from the operational
  notes table: policy denial, egress block, timeout/kill, and Azure
  errors.

### 5.1 — The policy

The agent's policy uses the native `PolicyDocument` schema. Every field —
rules, sandbox resource caps, network allowlist, and `network_default` —
lives in YAML; no `SimpleNamespace` wrapper is needed. A complete,
runnable version of this exact pattern lives at
[`examples/quickstart/azure_sandbox_step5_test.py`](../../examples/quickstart/azure_sandbox_step5_test.py).

```yaml
# policies/research_agent.yaml
name: research-agent
version: "2"

defaults:
  action: allow              # rule-engine default
  max_cpu: 1.0               # → sandbox resources.cpu = "1000m"
  max_memory_mb: 2048        # → sandbox resources.memory = "2048Mi"
  timeout_seconds: 90        # per-execute_code wall-clock cap
  network_default: deny      # fail-closed (this is the schema default too)

network_allowlist:
  - api.openai.com
  - api.arxiv.org
  - export.arxiv.org
  - "*.github.com"
  - pypi.org
  - files.pythonhosted.org

tool_allowlist:
  - fetch_arxiv
  - fetch_github_readme
  - search_index

# Each "code substring" pattern becomes its own deny rule because
# PolicyCondition takes a single `value`, not a list. Group related
# rules by giving them the same priority and message.
rules:
```

### 5.2 — The agent

```python
# research_agent.py
"""
Governed research agent that runs LLM-generated analysis code in an
Azure Container Apps sandbox under an AGT policy.

The agent never trusts the LLM. Every step it generates is:
  1. Gated by AGT PolicyDocument rules (host-side, before any Azure call).
  2. Executed inside a per-session Azure sandbox with CPU/memory caps and
     a fail-closed egress allowlist.
  3. Bounded by a per-step wall-clock timeout.
  4. Logged with a receipt that records the policy decision, the Azure
     sandbox id, and the egress decisions Azure made for the step.

Run with:
    AZURE_SUBSCRIPTION_ID=...   az login   python research_agent.py ticket.json
"""
from __future__ import annotations

import asyncio
import json
import logging
import os
import sys
import textwrap
import time
import uuid
from dataclasses import dataclass, field
from typing import Any

from agent_os.policies import PolicyDocument
from agent_sandbox import ACASandboxProvider
from openai import AsyncOpenAI

LOG = logging.getLogger("research-agent")
logging.basicConfig(level=logging.INFO, format="%(asctime)s %(levelname)s %(name)s %(message)s")

# ---------------------------------------------------------------------------
# 1. Ticket and step types
# ---------------------------------------------------------------------------

@dataclass
class Ticket:
    id: str
    title: str
    body: str

@dataclass
class Step:
    index: int
    intent: str          # short human description from the planner
    code: str            # python code to run in the sandbox

@dataclass
class StepReceipt:
    step_index: int
    intent: str
    decision: str        # "allowed" | "denied-by-policy" | "blocked-at-egress" | "timeout" | "error"
    reason: str | None
    azure_sandbox_id: str | None
    duration_seconds: float
    stdout_excerpt: str
    stderr_excerpt: str
    egress_decisions: list[dict] = field(default_factory=list)

# ---------------------------------------------------------------------------
# 2. Planner — asks the LLM for an ordered list of analysis steps
# ---------------------------------------------------------------------------

PLANNER_SYSTEM = """You are a research planner. Given a ticket, output JSON
of the form {"steps": [{"intent": str, "code": str}, ...]} where each `code`
is a self-contained Python snippet (no installs, no shell, no secrets).
Snippets may import only the Python standard library — the public sandbox
images do NOT preinstall third-party packages. Use `urllib.request` for
HTTP calls (not `requests`); `json`, `math`, `statistics`, `re`, `datetime`
are all available.
Snippets may reach: api.arxiv.org, export.arxiv.org, *.github.com, pypi.org.
Print structured JSON to stdout for the orchestrator to parse."""

def plan_steps(ticket: Ticket, model: str = "gpt-4o-mini") -> list[Step]:
    client = AsyncOpenAI()
    return _plan_steps_async(client, ticket, model)

async def _plan_steps_async(client: AsyncOpenAI, ticket: Ticket, model: str) -> list[Step]:
    resp = await client.chat.completions.create(
        model=model,
        response_format={"type": "json_object"},
        messages=[
            {"role": "system", "content": PLANNER_SYSTEM},
            {"role": "user", "content": json.dumps(
                {"ticket_id": ticket.id, "title": ticket.title, "body": ticket.body}
            )},
        ],
    )
    plan = json.loads(resp.choices[0].message.content)
    return [
        Step(index=i, intent=s["intent"], code=s["code"])
        for i, s in enumerate(plan.get("steps", []))
    ]

# ---------------------------------------------------------------------------
# 3. Orchestrator — runs each step under AGT + Azure sandbox
# ---------------------------------------------------------------------------

class ResearchAgent:
    """One instance per ticket. Owns one Azure sandbox session."""

    def __init__(
        self,
        provider: ACASandboxProvider,
        policy: Any,                    # SimpleNamespace wrapper around PolicyDocument
        agent_id: str,
    ) -> None:
        self._provider = provider
        self._policy = policy
        self._agent_id = agent_id
        self._handle = None
        self._sandbox_group = provider._sandbox_group  # needed for egress audit

    async def __aenter__(self) -> "ResearchAgent":
        self._handle = await self._provider.create_session_async(
            self._agent_id, policy=self._policy,
        )
        LOG.info("provisioned sandbox %s for agent %s",
                 self._handle.session_id, self._agent_id)
        return self

    async def __aexit__(self, *exc: Any) -> None:
        if self._handle is not None:
            try:
                await self._provider.destroy_session_async(
                    self._agent_id, self._handle.session_id,
                )
            except Exception:  # noqa: BLE001
                LOG.exception("destroy_session_async failed; sandbox may leak — "
                              "use Azure portal or `az sandbox delete` to clean up")

    # ----- per-step execution -----

    async def run_step(self, step: Step) -> StepReceipt:
        if self._handle is None:
            raise RuntimeError("agent not entered")

        started = time.monotonic()
        sid = self._handle.session_id

        # 1. Policy gate is *inside* execute_code_async; we just translate
        #    exceptions into structured receipts so the caller never has
        #    to catch.
        try:
            exec_handle = await self._provider.execute_code_async(
                self._agent_id,
                sid,
                step.code,
                context={"step_index": step.index, "intent": step.intent},
            )
        except PermissionError as exc:
            return StepReceipt(
                step_index=step.index,
                intent=step.intent,
                decision="denied-by-policy",
                reason=str(exc),
                azure_sandbox_id=sid,
                duration_seconds=time.monotonic() - started,
                stdout_excerpt="",
                stderr_excerpt="",
            )
        except Exception as exc:  # noqa: BLE001 — Azure / network errors
            return StepReceipt(
                step_index=step.index,
                intent=step.intent,
                decision="error",
                reason=f"{type(exc).__name__}: {exc}",
                azure_sandbox_id=sid,
                duration_seconds=time.monotonic() - started,
                stdout_excerpt="",
                stderr_excerpt="",
            )

        result = exec_handle.result

        # 2. Pull the Azure egress audit trail for this step. The data-plane
        #    SDK is sync today, so we hop to a worker thread to avoid blocking
        #    the event loop.
        egress = []
        try:
            decisions = await asyncio.to_thread(
                self._provider._data_client.get_egress_decisions,
                sid, self._sandbox_group,
            )
            egress = decisions.get("decisions", [])[-25:]  # last 25 decisions
        except Exception:  # noqa: BLE001
            LOG.debug("egress decision audit unavailable", exc_info=True)

        if getattr(result, "killed", False):
            decision = "timeout"
            reason = getattr(result, "kill_reason", "exceeded timeout_seconds")
        elif any(d.get("action") == "Deny" for d in egress):
            decision = "blocked-at-egress"
            reason = "Azure egress proxy denied at least one host"
        elif result.success:
            decision = "allowed"
            reason = None
        else:
            decision = "error"
            reason = f"exit code {result.exit_code}"

        return StepReceipt(
            step_index=step.index,
            intent=step.intent,
            decision=decision,
            reason=reason,
            azure_sandbox_id=sid,
            duration_seconds=time.monotonic() - started,
            stdout_excerpt=textwrap.shorten(result.stdout or "", width=512),
            stderr_excerpt=textwrap.shorten(result.stderr or "", width=512),
            egress_decisions=egress,
        )

# ---------------------------------------------------------------------------
# 4. Entry point
# ---------------------------------------------------------------------------

async def main(ticket_path: str) -> int:
    ticket = Ticket(**json.loads(open(ticket_path, encoding="utf-8").read()))
    # All sandbox fields are native to PolicyDocument — load and pass.
    policy = PolicyDocument.from_yaml("policies/research_agent.yaml")

    provider = ACASandboxProvider(
        resource_group=os.environ["AZURE_RG"],
        sandbox_group=os.environ.get("AZURE_SANDBOX_GROUP", "agents"),
        # Default "ubuntu" image has no python3; use a Python sandbox
        # image so execute_code's `… | python3` invocation works.
        disk=os.environ.get("AZURE_SANDBOX_DISK", "python-3.13"),
        ensure_group_location=os.environ.get("AZURE_REGION", "westus2"),
    )

    agent_id = f"research-{ticket.id}-{uuid.uuid4().hex[:6]}"
    receipts: list[StepReceipt] = []

    try:
        steps = await _plan_steps_async(AsyncOpenAI(), ticket, "gpt-4o-mini")
    except Exception as exc:  # noqa: BLE001
        LOG.error("planner failed: %s", exc)
        return 2

    LOG.info("planner produced %d steps for ticket %s", len(steps), ticket.id)

    async with ResearchAgent(provider, policy, agent_id) as agent:
        for step in steps:
            receipt = await agent.run_step(step)
            receipts.append(receipt)
            LOG.info("step %d (%s): %s%s",
                     step.index, step.intent, receipt.decision,
                     f" — {receipt.reason}" if receipt.reason else "")
            # Stop on the first non-recoverable decision so a bad step does
            # not trigger a cascade of retries against Azure.
            if receipt.decision in ("denied-by-policy", "blocked-at-egress", "timeout"):
                LOG.warning("halting plan after step %d", step.index)
                break

    # Persist the receipt bundle. In production this would go to an audit
    # backend (e.g. agent-governance-toolkit's audit pipeline) — the
    # in-process JSON file is a stand-in.
    out = f"receipts-{ticket.id}.json"
    with open(out, "w", encoding="utf-8") as f:
        json.dump(
            {
                "agent_id": agent_id,
                "ticket": ticket.__dict__,
                "policy_name": policy.name,
                "policy_version": policy.version,
                "receipts": [r.__dict__ for r in receipts],
            },
            f, indent=2, default=str,
        )
    LOG.info("wrote %s (%d receipts)", out, len(receipts))
    return 0 if all(r.decision == "allowed" for r in receipts) else 1

if __name__ == "__main__":
    sys.exit(asyncio.run(main(sys.argv[1])))
```

#### How `execute_code_async` runs the policy gate

The `try / except PermissionError` block in `run_step` is the *only*
place the orchestrator handles policy denials, because the gate itself
lives inside `ACASandboxProvider.execute_code` (which the async wrapper
calls via `asyncio.to_thread`). Walking through what happens between
`await provider.execute_code_async(...)` and the eventual
`PermissionError`:

1. **Evaluator is built once, at session creation.** During
   `create_session_async`, the provider does:

   ```python
   # aca_sandbox_provider.py — create_session (excerpt)
   if policy is not None:
       cfg = aca_config_from_policy(policy, cfg)
       allow_hosts = _network_allowlist(policy)
       try:
           from agent_os.policies.evaluator import PolicyEvaluator
           evaluator = PolicyEvaluator(policies=[policy])
       except ImportError:
           logger.warning(
               "agent-os-kernel not installed — "
               "policy evaluation unavailable, session runs ungated"
           )
   ...
   with self._state_lock:
       self._sandboxes[key] = sandbox_id
       self._session_configs[key] = cfg
       if evaluator is not None:
           self._evaluators[key] = evaluator
   ```

   The `PolicyEvaluator` is bound to `(agent_id, session_id)` in the
   provider's `_evaluators` dict so subsequent `execute_code` calls do
   not re-parse the policy YAML on every step.

2. **Every `execute_code` call looks the evaluator up and runs it
   *before* any Azure traffic is generated:**

   ```python
   # aca_sandbox_provider.py — execute_code (excerpt)
   def execute_code(self, agent_id, session_id, code, *, context=None):
       key = (agent_id, session_id)
       with self._state_lock:
           sandbox_id = self._sandboxes.get(key)
           evaluator  = self._evaluators.get(key)
           session_cfg = self._session_configs.get(key) or SandboxConfig()

       if sandbox_id is None:
           raise RuntimeError("No active session ...")

       if evaluator is not None:
           eval_ctx = {
               "agent_id": agent_id,
               "action":   "execute",
               "code":     code,
           }
           if context:
               eval_ctx.update(context)
           decision = evaluator.evaluate(eval_ctx)
           if not decision.allowed:
               raise PermissionError(f"Policy denied: {decision.reason}")

       # Only now do we base64-encode the source and call the data plane.
       encoded = base64.b64encode(code.encode("utf-8")).decode("ascii")
       resp = self._data_client.exec(
           sandbox_id, self._sandbox_group,
           f"echo {encoded} | base64 -d | python3",
           resource_group=self._resource_group,
       )
       ...
   ```

   The order is the security guarantee: `evaluator.evaluate(...)` runs on
   the host process, in the same thread that called `execute_code`. If
   the decision is `deny`, `PermissionError` is raised *before*
   `self._data_client.exec` is called, so no Azure data-plane request is
   issued, no sandbox CPU is consumed, and no egress decision is
   recorded for that step. From Azure's point of view the call simply
   never happened.

   ##### Inside `evaluator.evaluate(eval_ctx)`

   `eval_ctx` is just a flat `dict[str, Any]`. There is no special type,
   no schema validation at this point, no I/O. The evaluator
   ([`agent_os/policies/evaluator.py`](../../agent-governance-python/agent-os/src/agent_os/policies/evaluator.py))
   walks every rule from every loaded `PolicyDocument`, sorted by
   `priority` descending, and asks one question per rule: does this
   rule's `condition` match the dict?

   The match is a single field lookup followed by an operator comparison:

   ```python
   # evaluator.py — _match_condition (excerpt)
   def _match_condition(condition, context):
       ctx_value = context.get(condition.field)   # e.g. context["code"]
       if ctx_value is None:
           return False
       op = condition.operator
       target = condition.value
       if op == PolicyOperator.EQ:       return ctx_value == target
       if op == PolicyOperator.CONTAINS: return target in ctx_value
       if op == PolicyOperator.MATCHES:  return bool(re.search(str(target), str(ctx_value)))
       # ... NE, GT, LT, GTE, LTE, IN
   ```

   For the rule we wrote in [Step 5.1](#51--the-policy):

   ```yaml
   - name: deny-pip-install
     condition: { field: code, operator: contains, value: "pip install" }
     action: deny
     priority: 100
     message: "ad-hoc dependency installs are not permitted"
   ```

   `field: code` + `operator: contains` is how you gate snippets:
   the provider passes `eval_ctx = {"action": "execute", "code": <code>, ...}`,
   and the evaluator does a substring check on `eval_ctx["code"]`.
   Because `PolicyCondition.value` takes a single value, every
   substring you want to deny needs its own rule (group them by
   `priority` and `message` as in Step 5.1). When the orchestrator calls

   ```python
   eval_ctx = {
       "agent_id": "research-TKT-4821-9b2e1c",
       "action":   "execute",
       "code":     "import subprocess; subprocess.call('pip install requests', shell=True)",
       "step_index": 1,                    # from `context=`
       "intent":     "install dependency", # from `context=`
   }
   ```

   the evaluator iterates the rules:

   1. The first `deny-shell-out-*` rule (matching substring
      `"subprocess"`) checks `eval_ctx["code"]` → matches → returns
      immediately with
      `PolicyDecision(allowed=False, matched_rule="deny-shell-out-subprocess", reason="shell-out blocked …")`.
   2. The remaining rules never run, because evaluation short-circuits
      on the first match.

   The decision flows back into `execute_code`:

   ```python
   decision = evaluator.evaluate(eval_ctx)
   if not decision.allowed:
       raise PermissionError(f"Policy denied: {decision.reason}")
   ```

   `decision.reason` is the rule's `message` field (verbatim from YAML),
   which is what the receipt records as the human-readable explanation.
   `decision.audit_entry` (not used here, but available) carries a
   timestamped snapshot of `eval_ctx`, the matched rule name, and the
   policy name — that is what an audit pipeline would persist if you
   wanted full evidentiary records of *why* a step was denied.

   Three properties worth knowing:

   - **Field-by-field, no AST.** `operator: contains` is plain
     substring matching against `eval_ctx["code"]`. It does not parse
     the Python source, so an obfuscated payload like
     `getattr(__import__("subpro"+"cess"), "call")(...)` slips past a
     `value: "subprocess"` rule. The rule engine is a coarse first
     gate; defense in depth comes from the Azure egress proxy and the
     sandbox VM itself.
   - **Missing keys are not errors.** `context.get(condition.field)`
     returns `None` for unknown fields, and `_match_condition` returns
     `False`. A rule whose `field` references `step_index` against a
     context that has no `step_index` simply does not match — the
     evaluator does not raise.
   - **Fail-closed on internal error.** If `_match_condition` itself
     raises (a malformed regex, for example), the outer `evaluate`
     wrapper returns `PolicyDecision(allowed=False, action="deny",
     reason="Policy evaluation error — access denied (fail closed)")`
     so a broken rule cannot accidentally allow execution.

3. **`context` lets the agent enrich the evaluation.** The orchestrator
   in `run_step` passes:

   ```python
   exec_handle = await self._provider.execute_code_async(
       self._agent_id, sid, step.code,
       context={"step_index": step.index, "intent": step.intent},
   )
   ```

   Those values are merged into `eval_ctx` and become available to
   `rules:` in the policy. A rule like:

   ```yaml
   - name: deny-experimental-after-step-3
     condition: { field: step_index, operator: gte, value: 3 }
     action: deny
     priority: 200
     message: "experimental steps must run in a separate session"
   ```

   …can fire on the same code that would have been allowed at step 0
   (compose with a second rule on `intent contains "experimental"` if
   you want both conditions). This is how you express "trust decays
   over the life of a session" without changing the policy file
   mid-run.

4. **`agent-os-kernel` missing is *not* fatal.** If the import in step 1
   fails, `evaluator` stays `None`, the warning is logged once, and
   `execute_code` skips the gate entirely — the session runs ungated,
   matching the Docker backend's behavior. Production deployments should
   pin `agent-os-kernel` so this fallback never triggers; the
   [failure modes table](#failure-modes) below lists how to detect it.

5. **Async is purely a wrapper.** `execute_code_async` is the default
   from `SandboxProvider`:

   ```python
   async def execute_code_async(self, *args, **kwargs):
       return await asyncio.to_thread(self.execute_code, *args, **kwargs)
   ```

   The policy gate, the lock acquisition, and the data-plane call all
   happen inside the worker thread. From the event loop's perspective,
   `await provider.execute_code_async(...)` is one suspend point — the
   gate is not re-entrant and cannot be bypassed by cancelling the
   awaiting task, because cancellation does not propagate into the
   running thread.

The net effect for the orchestrator is that `run_step`'s
`except PermissionError` branch is the *complete* contract for
host-side policy denials. There is no other path by which a denied
snippet reaches Azure.

### 5.4 — What this example demonstrates

| Concern | How it is addressed |
|---------|---------------------|
| Untrusted LLM output | Every snippet is policy-gated *before* it leaves the host process; rules cover shell-out, package install, and credential reads |
| Resource abuse | `defaults.max_cpu` / `defaults.max_memory_mb` translate to sandbox CPU/memory; `timeout_seconds` is enforced wall-clock |
| Network exfiltration | `network_allowlist` becomes a fail-closed Azure egress policy; `get_egress_decisions` produces an independent audit trail |
| Cascading failures | Orchestrator halts on the first deny / block / timeout instead of retrying into rate limits |
| Forensics | Every step produces a structured `StepReceipt` with the policy decision, the Azure sandbox id, durations, and egress decisions |
| Resource cleanup | `ResearchAgent.__exit__` always calls `destroy_session`; provider's `delete_sandbox` is idempotent |
| Multi-tenancy | `agent_id` includes the ticket id and a fresh uuid, so concurrent tickets get independent sandboxes with independent egress allowlists |

This is the skeleton most production agents end up with: planner → policy
gate → sandbox execution → receipt. Replacing `ACASandboxProvider`
with the Docker or Hyperlight backend changes only the constructor call —
the rest of the orchestrator is identical.

## Step 6 — Drop down to the SDKs when you need to

The `SandboxProvider` ABC deliberately keeps a small surface. When you
need Azure-specific features that are *not* derivable from `policy` —
ad-hoc snapshots, port forwards, lifecycle policies, content packages,
egress audit retrieval outside `run_step` — reach the underlying SDKs
directly:

```python
import asyncio

from agent_sandbox import ACASandboxProvider

async def main(policy) -> None:
    provider = ACASandboxProvider(
        resource_group="agents-rg", sandbox_group="agents",
        disk="python-3.13",   # default "ubuntu" has no python3
    )
    handle = await provider.create_session_async("research", policy=policy)

    # The provider exposes the underlying clients for advanced use. They are
    # sync clients, so wrap them in `asyncio.to_thread` to keep the event
    # loop responsive.
    data_client = provider._data_client  # azure.sandbox.SandboxClient

    # Snapshot the running sandbox (Azure-native checkpoint).
    snap = await asyncio.to_thread(
        data_client.create_snapshot,
        handle.session_id, "agents", "post-pip-install",
    )

    # Audit the egress decisions Azure made on this sandbox.
    decisions = await asyncio.to_thread(
        data_client.get_egress_decisions,
        handle.session_id, "agents",
    )
    for d in decisions.get("decisions", []):
        print(d["host"], d["action"], d.get("reason"))

# Build `policy` as in Step 5.1, then run:
asyncio.run(main(policy))
```

This is escape hatch territory — code that uses `provider._data_client`
no longer benefits from the cross-backend portability the ABC provides.
A future revision of AGT plans to surface `commit_sandbox` /
`create_snapshot` through the optional `save_state` / `restore_state`
methods on the ABC; until then, the direct SDK is the supported path.

## Operational notes

### Cost control

`auto_suspend_seconds` defaults to 300 on `SandboxClient.create_sandbox`,
which means an idle sandbox is suspended after five minutes and the VM
stops billing for compute (memory remains charged). For bursty workloads
with many short sessions, prefer one short-lived sandbox per session
(`destroy_session` after each request) over keeping a warm pool.

### Failure modes

| Situation | What you see | What to do |
|-----------|--------------|------------|
| `azure-sandbox` not installed | `is_available()` returns `False`; `create_session` raises `RuntimeError` | `pip install azure-sandbox` |
| `agent-os-kernel` not installed | Provider logs a warning and runs ungated (matches Docker backend) | Pin `agent-os-kernel` if you need fail-closed |
| Invalid `agent_id` or `sandbox_group` | `ValueError` at `create_session` | Use `[a-zA-Z0-9][a-zA-Z0-9_-]{0,62}` |
| Resource group does not exist | `RuntimeError` from `create_session` wrapping Azure `ResourceGroupNotFound` 404 | Create the RG first: `az group create -n <rg> -l <region>`. The provider does not auto-create RGs. |
| Sandbox group does not exist | `HTTPStatusError` 404 from `create_sandbox` | Pass `ensure_group_location` or pre-create the group |
| Sandbox quota exceeded | `HTTPStatusError` 429 | Configure quota or destroy idle sessions |
| Code exceeds `cfg.timeout_seconds` | `SandboxResult.killed = True`, `kill_reason` set | Treat as a failed execution; the sandbox is still alive — call `destroy_session` |

## Step 7 — Pipe `StepReceipt` into AGT's audit hash-chain

The `StepReceipt` produced in [Step 5.2](#52--the-agent) is fine as a
local JSON blob, but for production you want **tamper-evident** receipts
that an auditor can verify offline. AGT ships two backends that take a
receipt-shaped record and chain it into a SHA-256 Merkle log:

| Backend | Where it lives | What you get |
|---------|----------------|--------------|
| `MerkleAuditChain` | [`agentmesh.governance.audit`](../../agent-governance-python/agent-mesh/src/agentmesh/governance/audit.py) | Append-only `AuditEntry` log with `previous_hash` + `entry_hash` + Merkle root |
| `GovernanceReceipt` | [`mcp_receipt_governed.receipt`](../../agent-governance-python/agentmesh-integrations/mcp-receipt-governed/mcp_receipt_governed/receipt.py) | Per-call signed receipts with `parent_receipt_hash`, Ed25519 signature, SLSA provenance |

Use `MerkleAuditChain` when you want a single tamper-evident log per
agent run; use `GovernanceReceipt` when each step needs to be
independently signed and shipped (e.g. across trust boundaries). Both
expect an entry that names the actor, the action, the outcome, and any
matched policy decision — which is exactly what `StepReceipt` carries.

A `StepReceipt` → `AuditEntry` adapter is six lines:

```python
from agentmesh.governance.audit import AuditEntry, MerkleAuditChain

chain = MerkleAuditChain()  # one per ticket / per agent run

def chain_receipt(receipt: StepReceipt, agent_id: str, ticket_id: str) -> AuditEntry:
    entry = AuditEntry(
        event_type="sandbox.execute",
        agent_did=agent_id,
        action=f"step:{receipt.step_index}",
        resource=f"ticket:{ticket_id}",
        data={
            "intent": receipt.intent,
            "azure_sandbox_id": receipt.azure_sandbox_id,
            "duration_seconds": receipt.duration_seconds,
            "egress_decisions": receipt.egress_decisions,
            "stdout_excerpt": receipt.stdout_excerpt,
            "stderr_excerpt": receipt.stderr_excerpt,
        },
        outcome=receipt.decision,             # allowed | denied-by-policy | ...
        policy_decision=receipt.decision,
        matched_rule=receipt.reason,
        session_id=receipt.azure_sandbox_id,
    )
    chain.add_entry(entry)                    # sets previous_hash + entry_hash
    return entry
```

Wire it into the orchestrator by calling `chain_receipt(receipt, ...)`
immediately after `receipts.append(receipt)` in `main()`. At the end of
the run, persist `chain.get_root_hash()` alongside the receipt bundle
— that single hex digest is what an auditor needs to verify the entire
run was not tampered with.

If you prefer signed per-step receipts over a single Merkle log, the
`GovernanceReceipt` path is structurally identical: build one receipt
per step, set `parent_receipt_hash` to the prior step's hash, sign with
an Ed25519 key, and ship to your audit sink. See
[`mcp_receipt_governed.receipt.verify_receipt_chain`](../../agent-governance-python/agentmesh-integrations/mcp-receipt-governed/mcp_receipt_governed/receipt.py)
for the verification side.

> **External exporters.** Citadel, OpenTelemetry, or any SIEM that
> accepts a JSON event stream can sit downstream of either backend —
> `AuditEntry` and `GovernanceReceipt` both serialize cleanly via
> `.model_dump_json()`. Treat the AGT chain as the source of truth and
> the exporter as a delivery channel, not a replacement.

## What's next — Part 2

This article stops where most production deployments start: a governed
sandbox that runs untrusted code, denies bad calls fail-closed, and
emits tamper-evident receipts. The next layer of governance is **how
those receipts feed back into trust**.

### Trust scoring meets sandbox lifecycle

The receipts produced in [Step 7](#step-7--pipe-stepreceipt-into-agts-audit-hash-chain)
are not just an audit artifact — they are the **input signal** to AGT's
trust engine. Each `StepReceipt.decision` (`allowed` / `denied-by-policy` /
`blocked-at-egress` / `timeout` / `error`) is a graded outcome the trust
engine can score: clean runs raise the agent's score, denials and egress
blocks decay it, and the Merkle root makes the entire history
non-repudiable.


A follow-up article will cover:

- **Trust scoring from receipts** — how AGT's trust engine consumes the
  audit chain to adjust an agent's reputation over time, and what
  policy operators (`trust_score gte 0.8`) look like in
  `PolicyDocument`.
- **Sandbox lifecycle hooks** — `save_state` / `restore_state` for
  snapshotting a sandbox at a known-good receipt, rolling back on a
  policy violation, and re-running steps under a tightened policy
  without losing prior state.
- **Trust decay with `step_index`** — concrete policies that ratchet
  privileges down as a session ages, using receipt-derived signals.

If you have a use case that needs any of those today, open a discussion
on the [AGT repo](https://github.com/microsoft/agent-governance-toolkit) — the
trust-engine surface is being shaped by real workloads, and integration
patterns are still in flux.
