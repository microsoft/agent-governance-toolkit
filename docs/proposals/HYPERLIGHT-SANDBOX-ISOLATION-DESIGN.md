## Motivation

`DockerSandboxProvider` gives `agent-sandbox` a hardened container backend,
but container isolation is fundamentally **kernel-shared**. A kernel exploit,
a container-escape CVE, or a misconfigured runtime breaks the boundary. For
high-risk (Ring 3) agents — third-party plugins, marketplace agents, and code
generated by an untrusted upstream — we need a sandbox where the guest cannot
reach the host kernel at all.

[Hyperlight](https://github.com/hyperlight-dev/hyperlight) is a lightweight
Virtual Machine Manager (VMM) that runs guest code in **micro virtual
machines** with hardware-level isolation via KVM (Linux), Microsoft Hypervisor
(mshv on Azure Linux), or Windows Hypervisor Platform (WHP on Windows 11 /
Server 2022+). Each guest runs without a general-purpose OS and without a
filesystem.

[`hyperlight-sandbox`](https://github.com/hyperlight-dev/hyperlight-sandbox)
is a separate CNCF Sandbox project (Apache-2.0) that wraps Hyperlight with
a unified, multi-backend sandboxing API and ships official Python and .NET
SDKs. It provides everything `HyperLightSandboxProvider` needs out of the box:

* a Python SDK (`pip install "hyperlight-sandbox[wasm,python_guest]"`),
* a packaged Python guest that runs **real Python source** inside a Wasm
  component on hyperlight-wasm (no custom guest binary required),
* `register_tool(name, callable)` for tool dispatch with schema-validated
  args,
* `allow_domain(url)` plus a built-in `http_get` / `fetch` exposed to the
  guest only after a domain is allowed,
* `snapshot()` / `restore(snap)` for in-memory rewind, and
* read-only `/input` and writable `/output` capability-based file access.

The right move for AGT is to **adopt `hyperlight-sandbox` as the underlying
runtime** and write `HyperLightSandboxProvider` as a thin adapter from the
`SandboxProvider` ABC defined in
[DOCKER-SANDBOX-ISOLATION-DESIGN.md](./DOCKER-SANDBOX-ISOLATION-DESIGN.md).
Rebuilding what upstream already ships would duplicate effort and fall
behind on security fixes.

`HyperLightSandboxProvider` implements the same `SandboxProvider` ABC as
`DockerSandboxProvider` and `AzureSandboxProvider`, so callers can swap
backends without changing application code.

## Generic SandboxProvider Interface (Recap)

The `SandboxProvider` ABC and its `create_session` / `execute_code` /
`destroy_session` lifecycle are defined in
[DOCKER-SANDBOX-ISOLATION-DESIGN.md](./DOCKER-SANDBOX-ISOLATION-DESIGN.md#generic-sandboxprovider-interface).
The returned `SessionHandle` and `ExecutionHandle` types, status enums, and
async defaults are reused unchanged.

| Method | `HyperLightSandboxProvider` behavior |
|--------|--------------------------------------|
| `create_session(agent_id, policy, config)` | Constructs a `hyperlight_sandbox.Sandbox` for the chosen backend (default: Wasm + packaged Python guest), calls `register_tool` only for names in `policy.tool_allowlist`, calls `allow_domain` for each entry in `policy.network_allowlist`, mounts `/input` / `/output` capability directories per `config`, and stores the sandbox for the session. |
| `execute_code(agent_id, session_id, code, *, context)` | Evaluates the policy, then forwards `code` to `Sandbox.run(code)`. The guest is responsible for calling `call_tool(name, **args)` and `http_get(url)` as needed. Returns stdout/stderr in a `SandboxResult`. |
| `destroy_session(agent_id, session_id)` | Drops the `Sandbox` (Python `__del__` releases the underlying VM); removes the session entry and the policy evaluator. |
| `is_available()` | `True` if `hyperlight_sandbox` imports successfully and a hypervisor (KVM, mshv, or WHP) is reachable. |
| `save_state` / `restore_state` / `list_checkpoints` / `delete_checkpoint` | Not implemented. Hyperlight has no filesystem to commit. Use `snapshot_session` / `restore_snapshot` instead — see [Snapshots](#snapshots). |

Code written against the `SandboxProvider` ABC works unchanged on Hyperlight.
Docker-only methods raise `NotImplementedError` on this provider; callers
that depend on them must check `isinstance(provider, DockerSandboxProvider)`
first.

## DockerSandboxProvider vs HyperLightSandboxProvider

| Aspect | Docker | Hyperlight (`hyperlight-sandbox`) |
|--------|--------|-----------------------------------|
| Isolation | Container (runc); optional gVisor / Kata | Hardware VM (KVM / mshv / WHP) wrapping a Wasm component, HyperlightJS, or Nanvix guest |
| Kernel | Shared with host (or guest kernel for Kata) | **No general-purpose OS in the guest**; the guest sees only the Wasm component runtime / HyperlightJS runtime / Nanvix microkernel |
| Filesystem in guest | Full Linux FS (read-only root + tmpfs) | Capability-based: read-only `/input`, writable `/output`, no other paths |
| Code model | `python -c "…"` in container | `Sandbox.run("…")` runs Python source on the packaged Python guest, or JS source on the JS guest |
| Tool calls | HTTP proxy on host port 9100 | `Sandbox.register_tool(name, callable)`; guest calls `call_tool(name, **args)` with schema-validated args, brokered through the hypervisor |
| Network | HTTP proxy + iptables (Linux) / proxy-only (macOS, Windows) | `Sandbox.allow_domain(url)`; guest calls upstream-provided `http_get` / `fetch`. No network at all if no domain allowed. |
| Resource limits | cgroups (`mem_limit`, `nano_cpus`) | Hyperlight `SandboxConfiguration` (heap, stack, max_execution_time) — set by upstream; AGT exposes the knobs via `HyperlightConfig` |
| State | Persists across `exec_run` calls; `docker commit` snapshots | Persists across `Sandbox.run` calls; in-memory `snapshot()` / `restore(snap)` (CoW) |
| Cold start | 100–500 ms | **1–10 ms** for empty Wasm guest; ~50 ms for the Python guest's first warmup, then 1–10 ms |
| Memory overhead | 50–200 MB per container | 1–10 MB for the runtime; the Python guest adds the size of the embedded CPython component |
| Platform | Linux native; macOS/Windows via Docker Desktop Linux VM | Linux (KVM or mshv); Windows 11 / Server 2022+ (WHP); **macOS not supported** |
| License of the runtime | Apache-2.0 (`docker`/`runc`) | Apache-2.0 (`hyperlight`, `hyperlight-sandbox`) |

## Goals

1. **Implement `SandboxProvider` on `hyperlight-sandbox`** so callers can run
   high-risk agents with hardware-level isolation through the same API as
   the Docker backend.
2. **Map `tool_allowlist` directly to `register_tool` calls.** Only host
   functions named in the allowlist are registered; tools registered with
   the provider but not allowlisted are not handed to the guest at all.
3. **Map `network_allowlist` to `allow_domain` calls.** No allowed domains
   ⇒ guest has zero network capability — `http_get` / `fetch` will refuse
   every URL because the allowlist is empty.
4. **Pass policy `defaults.timeout_seconds` and resource caps** through to
   Hyperlight's `max_execution_time` / heap / stack via the upstream
   `Sandbox` builder.
5. **Preserve session state** across calls within a single `Sandbox`,
   matching the Docker provider's "container reused across `execute_code`
   calls" model.
6. **Expose snapshots** as provider-specific `snapshot_session` /
   `restore_snapshot` methods — not on the ABC.
7. **Support all three upstream backends** via a `backend=` parameter:
   `"wasm"` (default; runs Python or JS), `"hyperlightjs"` (JS-only,
   smaller footprint), `"nanvix"` (limited preview — no tools, FS,
   network, or snapshots).

## Non-Goals

- Running arbitrary shell commands or starting subprocesses inside the
  sandbox. The packaged Python guest is a sandboxed CPython interpreter; it
  does not have `subprocess`, `os.system`, or a process model. Use tools
  (`register_tool`) for anything beyond pure-Python computation.
- macOS support. Hyperlight requires KVM, mshv, or WHP. macOS users should
  run AGT in a Linux VM or use the Docker provider.
- Building or shipping our own guest binaries. Use the packaged Python
  guest, the packaged JS guest, or — for advanced users — a custom Wasm
  component, but this provider does not own a guest build.
- Filesystem checkpointing. The guest's filesystem is `/input` (read-only)
  and `/output` (per-session, ephemeral). Snapshots capture VM memory only.
- Using the lower-level `hyperlight` host crate directly. The provider
  defers to `hyperlight-sandbox` so AGT rides upstream security fixes and
  feature work rather than maintaining its own host integration.

## Summary

`HyperLightSandboxProvider` implements the `SandboxProvider` ABC by adapting
upstream `hyperlight_sandbox.Sandbox`. Tool calls and network access are
both delivered as **registered capabilities at session-construction time**.
The guest can only reach a tool if it was passed to `register_tool` before
construction; it can only reach a domain if it was passed to `allow_domain`
before construction. Capabilities the policy does not grant **do not exist**
in the guest's call vocabulary at runtime — the strongest form of policy
enforcement available.

| Capability | Detail |
|------------|--------|
| Isolation | Hardware-virtualized micro-VM per session (KVM / mshv / WHP) |
| Tool enforcement | `Sandbox.register_tool(name, callable)` per policy `tool_allowlist`; tools registered with the provider but not in the allowlist are not handed to the guest |
| Network enforcement | `Sandbox.allow_domain(url)` per policy `network_allowlist`; if the allowlist is empty, `http_get` and `fetch` refuse every URL |
| Code surface in guest | Real Python source (Wasm + packaged Python guest) or JS source (HyperlightJS / Wasm + JS guest); no shell, no subprocess, no filesystem outside `/input` and `/output` |
| State | In-memory; persists across `Sandbox.run` calls; optional `snapshot()` / `restore()` |
| Session lifecycle | One `Sandbox` per session; session ends when the Python object is dropped |
| Platform support | Linux (KVM or mshv); Windows 11 / Server 2022+ (WHP); no macOS |

### Crash isolation

A guest fault traps to the hypervisor, which returns a Rust error to
`Sandbox.run`, which raises an exception that AGT catches in
`execute_code`. The host Python process is never left in a corrupt state
by a misbehaving guest, so the provider does not need an out-of-process
runner for fault containment.

### Backend selection

```python
HyperLightSandboxProvider(backend="wasm",      module="python_guest")  # default
HyperLightSandboxProvider(backend="wasm",      module="javascript_guest")
HyperLightSandboxProvider(backend="hyperlightjs")
HyperLightSandboxProvider(backend="nanvix",    language="python")
```

| Backend | Guest languages | Tools | FS (`/input`, `/output`) | Network | Snapshots | When to use |
|---------|-----------------|-------|--------------------------|---------|-----------|-------------|
| `wasm` (default) | Python, JS, custom Wasm | ✅ | ✅ | ✅ | ✅ | The default for AGT — everything works |
| `hyperlightjs` | JS only | ✅ | ✅ | ✅ | ✅ | JS-only workloads with smaller footprint |
| `nanvix` | Python or JS | ❌ | ❌ | ❌ | ❌ | Preview only; raw `print` is the only side effect — useful for benchmarking and microkernel research |

The provider raises `ValueError` at `__init__` if a backend is requested but
its upstream extra is not installed (`hyperlight-sandbox[wasm,python_guest]`,
`hyperlight-sandbox[wasm,js_guest]`, `hyperlight-sandbox[hyperlightjs]`,
`hyperlight-sandbox[nanvix]`).

### Code Argument Semantics

`SandboxProvider.execute_code(code, *, context)` takes a string `code`
argument. On Docker this is Python source executed by `python -c`. On
Hyperlight it is the same — Python source for the `wasm` + `python_guest`
backend, JavaScript source for the JS backends. The provider forwards
`code` to upstream's `Sandbox.run(code)` unchanged. With
`hyperlight-sandbox`'s packaged Python guest the guest *is* a Python
interpreter, so the natural contract for `execute_code` matches Docker's
and callers do not need to know they are on a different backend.

The `context` kwarg is passed through `Sandbox.run`'s globals so the guest
can read it via `globals()["context"]`. AGT does **not** mutate `code` to
inject a context binding — that would change the source the user gave us
and confuse policy auditors.

### Integration with agent-os Policy

```
PolicyDocument (YAML / runtime object)
  ├── rules[]            → PolicyEvaluator; evaluated on every execute_code
  ├── defaults.max_cpu / max_memory_mb / timeout_seconds
  │                      → HyperlightConfig (heap, stack, max_execution_time)
  ├── tool_allowlist     → Sandbox.register_tool(name, callable) for each
  │                        allowlisted name found in self._tools
  └── network_allowlist  → Sandbox.allow_domain(url) for each entry;
                           empty / absent ⇒ guest has no network capability
```

When `create_session` receives a `PolicyDocument` (or the SimpleNamespace
wrapper described in the Azure backend doc):

1. Resource limits are translated to Hyperlight's `SandboxConfiguration`
   knobs by the upstream Sandbox builder.
2. The provider iterates `policy.tool_allowlist`. For each name found in
   `self._tools`, it calls `sandbox.register_tool(name, self._tools[name])`.
   Names listed in the allowlist that are not in `self._tools` raise
   `ValueError` — fail closed at session creation, not call time.
3. The provider iterates `policy.network_allowlist`. For each entry it
   calls `sandbox.allow_domain(entry)`. If the allowlist is empty or
   missing, no `allow_domain` calls are made; the guest's `http_get`
   refuses every URL.
4. A `PolicyEvaluator` is bound to `(agent_id, session_id)` and re-used
   on every `execute_code` call.

```python
from agent_os.policies import PolicyDocument
from agent_sandbox import HyperLightSandboxProvider

policy = PolicyDocument.from_yaml("policies/research_policy.yaml")
provider = HyperLightSandboxProvider(
    backend="wasm",
    module="python_guest",
    tools={
        "web_search": lambda query: search(query),
        "read_doc":   lambda doc_id: load_doc(doc_id),
        # delete_doc is registered with the provider but NOT in the
        # policy's tool_allowlist → it is not handed to the guest
        "delete_doc": lambda doc_id: delete_doc(doc_id),
    },
)

handle = provider.create_session("research-agent", policy=policy)

exec_handle = provider.execute_code(
    "research-agent",
    handle.session_id,
    code=(
        "summary = call_tool('web_search', query='hyperlight')\n"
        "doc     = call_tool('read_doc',   doc_id='2401.12345')\n"
        "abstract = http_get('https://api.arxiv.org/abs/2401.12345')['body']\n"
        "print(summary[:200], len(abstract))\n"
    ),
)
print(exec_handle.result.stdout)

provider.destroy_session("research-agent", handle.session_id)
```

## How `HyperLightSandboxProvider` Implements Each Method

### `create_session(agent_id, policy, config) → SessionHandle`

1. **Hypervisor / SDK check.** If `is_available()` is false, raise
   `RuntimeError("hyperlight-sandbox unavailable: install the SDK and "
   "ensure a hypervisor (KVM / mshv / WHP) is reachable")`.
2. **`session_id`** = `uuid4().hex[:8]`.
3. **Resolve resource caps** from `policy.defaults` (`max_cpu` is ignored
   on Hyperlight — one vCPU per micro-VM; `max_memory_mb` / `timeout_seconds`
   become heap-budget and `max_execution_time`).
4. **Resolve `tool_allowlist`.**
   `allowed = {n: self._tools[n] for n in policy.tool_allowlist if n in self._tools}`.
   Missing names raise `ValueError`.
5. **Resolve `network_allowlist`.** Capture the list verbatim. An empty
   list disables network entirely (no `allow_domain` calls).
6. **Construct the upstream Sandbox.**

   ```python
   sandbox = hyperlight_sandbox.Sandbox(
       backend=self._backend,
       module=self._module,
       max_execution_time_ms=cfg.timeout_seconds * 1000,
       heap_size_bytes=cfg.max_memory_mb * 1024 * 1024,
   )
   for name, fn in allowed.items():
       sandbox.register_tool(name, fn)
   for url in net_allow:
       sandbox.allow_domain(url)
   ```

7. **Bind a `PolicyEvaluator`.**
   `self._evaluators[(agent_id, session_id)] = PolicyEvaluator(policies=[policy])`.
8. **Return** `SessionHandle(agent_id, session_id, status=READY)`.

### `execute_code(agent_id, session_id, code, *, context) → ExecutionHandle`

1. **Session lookup.** `(agent_id, session_id)` must be in `_sandboxes`,
   otherwise `RuntimeError("No active session — call create_session() first")`.
2. **Policy gate.** Evaluate the bound `PolicyEvaluator`. On deny, raise
   `PermissionError` *before* any guest call.
3. **Run the guest.**

   ```python
   try:
       result = sandbox.run(code, globals={"context": context or {}})
   except hyperlight_sandbox.GuestTrap as exc:
       return ExecutionHandle(... status=FAILED,
           result=SandboxResult(success=False, exit_code=1,
                                stderr=str(exc), ...))
   ```

   `Sandbox.run` returns an object with `stdout`, `stderr`, and
   `duration_ms` attributes. We wrap them in `SandboxResult` exactly like
   the Docker provider.
4. **Wall-clock kill.** If `result.duration_ms / 1000 > cfg.timeout_seconds`,
   set `result.killed = True`. The hypervisor already tore down the call —
   this only flags it for the caller.
5. **Return** `ExecutionHandle(..., status=COMPLETED|FAILED, result=...)`.

### `destroy_session(agent_id, session_id) → None`

1. Pop the `Sandbox` and the `PolicyEvaluator` from the bookkeeping dicts.
2. Drop the `Sandbox` reference; upstream releases the underlying VM.
3. Idempotent — calling on an unknown session is a no-op (matches the
   Azure provider).

### `is_available() → bool`

`True` iff:

* `import hyperlight_sandbox` succeeds, **and**
* `hyperlight_sandbox.is_hypervisor_present()` returns `True`
  (KVM, mshv, or WHP reachable), **and**
* the requested backend's optional dependency is installed (e.g. the
  packaged Python guest for `backend="wasm", module="python_guest"`).

### Snapshots (provider-specific, not on the ABC)

```python
snap = provider.snapshot_session("research-agent", session_id)
# … run more code, possibly diverge …
provider.restore_snapshot("research-agent", session_id, snap.snapshot_id)
```

`snapshot_session` calls `Sandbox.snapshot()`; `restore_snapshot` calls
`Sandbox.restore(snap)`. Snapshots live in process memory; destroying the
session drops them.

## Tool-Call Restriction (Host Functions)

Hyperlight's call mechanism makes the policy's `tool_allowlist` an integral
part of how a tool becomes callable, not a check applied later.

### Architecture

```
┌──────────────────────────────────────────────────┐
│  Hyperlight Guest VM (Wasm + Python guest)        │
│                                                  │
│  user code:                                      │
│    summary = call_tool('web_search',             │
│                        query='hyperlight')       │
│         │                                        │
│         │ hypercall (flatbuffer payload)         │
│         ▼                                        │
└─────────┼────────────────────────────────────────┘
          │
          ▼  (hypervisor-mediated call_tool dispatch)
┌──────────────────────────────────────────────────┐
│  hyperlight-sandbox host bindings                 │
│                                                  │
│  Tools registered for THIS session (per policy): │
│    web_search(query) -> bytes                    │
│    read_doc(doc_id) -> bytes                     │
│                                                  │
│  NOT registered:                                 │
│    delete_doc(...)   ← uncallable                │
│    fs_read(path)     ← uncallable                │
│    spawn_process(..) ← uncallable                │
└─────────┬────────────────────────────────────────┘
          │  Python callback into self._tools[name]
          ▼
┌──────────────────────────────────────────────────┐
│  HyperLightSandboxProvider                       │
│  Returns the Python callable's result            │
│  (schema-validated by upstream before dispatch)  │
└──────────────────────────────────────────────────┘
```

### Why this is stronger than the Docker `ToolCallProxy`

| Property | Docker `ToolCallProxy` | Hyperlight `register_tool` |
|----------|------------------------|----------------------------|
| Where the allowlist is enforced | HTTP server on host port 9100 | Hypervisor call gate, decided at `Sandbox` construction |
| What happens if the agent ignores the tool API | Could attempt raw sockets (blocked by iptables on Linux; bypassable on macOS/Windows) | The agent has no symbol to call. There is no "raw sockets" alternative — there is no kernel and no syscall. |
| Allowlist mutation at runtime | `proxy.register(ip, allowlist)` — a bug in the proxy could grant unintended tools | Tool set is fixed at construction; not mutable for the life of the session |
| Trusted computing base | `ToolCallProxy` (Python), HTTP parser, iptables | Hyperlight host crate, hypervisor |

A session created without a `tool_allowlist` registers no tools. The guest
can still execute code — `print`, arithmetic, the standard library — but it
has no way to reach Python callables on the host.

## Network Domain Restriction

Hyperlight guests have no general-purpose network stack. Network egress can
only occur through `http_get` / `fetch`, which upstream registers
automatically and which validates every URL against the allowlist passed
to `allow_domain`.

### Architecture

```
┌──────────────────────────────────────────────────┐
│  Hyperlight Guest VM (no socket syscalls)         │
│                                                  │
│  resp = http_get(                                │
│      "https://api.arxiv.org/abs/2401.12345")     │
│       │                                          │
│       │ hypercall                                │
│       ▼                                          │
└───────┼──────────────────────────────────────────┘
        │
        ▼
┌──────────────────────────────────────────────────┐
│  hyperlight-sandbox http_get implementation       │
│                                                  │
│  1. Parse url → "api.arxiv.org"                  │
│  2. Match against this Sandbox's allow list      │
│     (subdomain rule applied by upstream)         │
│     ├─ deny  → return {status: 403, ...}         │
│     └─ allow → reqwest::Client::get(url)         │
│  3. Stream response body back as bytes           │
└──────────────────────────────────────────────────┘
```

### Allowlist rules (handled by upstream)

| Rule | Behavior |
|------|----------|
| Exact match | `allow_domain("https://api.arxiv.org")` allows requests to that exact host |
| Subdomain | upstream's matcher accepts subdomain expansions (e.g. `arxiv.org` allows `api.arxiv.org`) |
| Scheme | `https://` is the default; `http://` requires an explicit `allow_domain("http://…")` entry |
| Method | All standard HTTP methods are supported; the policy can restrict via tool wrappers if tighter control is needed |
| Body / header sizes | Bounded by Hyperlight's input/output buffer config |

### Why this is un-bypassable

| Bypass technique on Docker | Status on Hyperlight |
|----------------------------|----------------------|
| Unset `HTTP_PROXY` env var | Irrelevant — there is no proxy and no env-driven HTTP client in the guest |
| Open a raw TCP socket | Irrelevant — there is no socket syscall and no kernel |
| `curl` directly to an IP | Irrelevant — there is no `curl`, no shell, no process model |
| DNS lookup to internal IP | Irrelevant — the host parses and validates the URL before any DNS happens |
| Spawn a child process | Irrelevant — guest cannot create processes |

Network policy bypass on Hyperlight requires either a `hyperlight-sandbox`
host-side vulnerability or a hypervisor escape (KVM / mshv / WHP). Both
are escalations of severity an order of magnitude beyond a typical Docker
bypass.

A session created without a `network_allowlist` makes zero `allow_domain`
calls. The guest can still call `http_get`, but every URL is denied at the
host before any network I/O happens.

## Internal Implementation Details

### Upstream APIs Used

| Step | `hyperlight_sandbox` API | Purpose |
|------|--------------------------|---------|
| Probe runtime | `is_hypervisor_present()` | Detect KVM / mshv / WHP availability |
| Construct sandbox | `Sandbox(backend=…, module=…)` | Build a per-session VM |
| Register tool | `sandbox.register_tool(name, callable)` | One call per allowlisted tool |
| Allow domain | `sandbox.allow_domain(url)` | One call per `network_allowlist` entry |
| Mount input dir | `sandbox.mount_input(path)` | Read-only `/input` |
| Mount output dir | `sandbox.mount_output(path)` | Per-session writable `/output` |
| Run code | `sandbox.run(source) -> RunResult` | Execute Python (or JS) source |
| Snapshot | `sandbox.snapshot() -> Snapshot` | Capture VM memory (CoW) |
| Restore | `sandbox.restore(snapshot)` | Restore VM memory |
| Cancel | `sandbox.interrupt()` | Trigger `max_execution_time` termination |

### `HyperlightConfig` Mapping

```python
@dataclass
class HyperlightConfig:
    backend: Literal["wasm", "hyperlightjs", "nanvix"] = "wasm"
    module:  str = "python_guest"        # ignored for hyperlightjs/nanvix
    heap_size_bytes:        int = 64 * 1024 * 1024
    stack_size_bytes:       int = 2  * 1024 * 1024
    max_execution_time_ms:  int = 60_000
    input_dir:  Path | None = None       # mounted read-only as /input
    output_dir: Path | None = None       # mounted writable as /output
```

`max_execution_time_ms` enforces a per-call timeout at the hypervisor
boundary — the guest is unconditionally torn out of execution when it
expires. This is the analogue of the Docker provider's per-call timeout,
but enforced by the hypervisor rather than by the SDK.

### Internal Data Structures

```
HyperLightSandboxProvider
├── _backend: Literal["wasm","hyperlightjs","nanvix"]
├── _module: str | None
├── _tools: dict[str, Callable]                            # provider-registered callables
├── _available: bool                                       # hypervisor + SDK detected at __init__
├── _sandboxes: dict[tuple[str,str], hyperlight_sandbox.Sandbox]
├── _evaluators: dict[tuple[str,str], PolicyEvaluator]
└── _state_lock: threading.Lock
```

All guest dispatch happens in-process through the upstream Python SDK, so
the provider does not need its own runner subprocess, host-call thread,
or RPC framing — `_sandboxes` and `_evaluators` are the entire bookkeeping
surface.

### Request Flow: `execute_code()` End-to-End

```
caller
  │
  ▼
provider.execute_code_async(agent_id="a1", session_id="abc12345",
                            code="…python source…",
                            context={...})
  │
  ├─ 0. Session check
  │     └─ ("a1", "abc12345") not in _sandboxes
  │        → raise RuntimeError("No active session — call create_session() first")
  │
  ├─ 1. Policy gate (host-side)
  │     ├─ _evaluators[("a1","abc12345")] exists
  │     │  → evaluator.evaluate({"agent_id":"a1","action":"execute","code":code,...})
  │     │     ├─ denied  → raise PermissionError  (no guest traffic)
  │     │     └─ allowed → continue
  │     └─ No evaluator → skip
  │
  ├─ 2. sandbox.run(code, globals={"context": context or {}})
  │     ├─ guest may call call_tool(name, **kw) → upstream dispatches
  │     │   to self._tools[name] iff registered
  │     ├─ guest may call http_get(url)        → upstream allows iff
  │     │   url.host ∈ allow_domain set
  │     ├─ guest may write to /output (if mounted)
  │     └─ returns RunResult(stdout, stderr, duration_ms, exit_code)
  │
  └─ 3. Wrap in ExecutionHandle / SandboxResult
        ├─ honor cfg.timeout_seconds (set killed=True if exceeded)
        └─ truncate stdout/stderr to 10 KiB
```

## Comparison to the Azure Sandbox provider

`HyperLightSandboxProvider` and `AzureSandboxProvider` solve different
points in the deployment matrix:

| Concern | Azure | Hyperlight |
|---------|-------|------------|
| Where the sandbox runs | Azure Container Apps (managed) | The agent's host machine |
| Who provisions the VM | Microsoft | The agent runtime |
| Cold-start cost | seconds (Azure provisioning) | milliseconds |
| Steady-state latency | ~100 ms (data-plane round trip) | microseconds (in-process) |
| Best for | Multi-tenant production agents and bursty workloads | Local dev, edge, on-prem, latency-sensitive control loops |
| Network policy enforcement | Azure egress proxy (HTTP 403) | Upstream `allow_domain` (host-side reject) |
| Tool isolation | None — host-side `PolicyEvaluator` only | Hypervisor call gate |

Both implement the same `SandboxProvider` ABC, so an application can switch
between them by changing the constructor.

## Risks & open questions

* **Upstream API stability.** `hyperlight-sandbox` is at v0.4.0 (CNCF
  Sandbox project). The API is moving but the surface used here
  (`register_tool`, `allow_domain`, `run`, `snapshot`/`restore`) has been
  stable across the v0.3 → v0.4 jump. We should pin a tight version range
  and bump deliberately, mirroring our policy for `azure-sandbox`.
* **Wasm Python guest size.** The packaged Python guest carries a CPython
  build inside the Wasm component. First-run materialization can be tens
  of MiB. This is a one-time cost, but worth documenting for air-gapped
  environments.
* **Tool argument schemas.** Upstream validates tool args against a schema
  derived from the registered Python callable's type hints. Untyped
  callables fall back to a permissive schema. AGT should require type
  hints (or explicit `inspect.signature(...)` decoration) for any tool
  that crosses the trust boundary.
* **`context` propagation.** Passing `context` via `globals()` is a
  pragmatic choice; if upstream adds first-class context support to
  `Sandbox.run` we should switch.
* **macOS.** Still no path. Document the gap and suggest WSL or remote
  Linux dev for macOS-based contributors.
* **Nanvix backend.** Currently a preview with no tools, no FS, no
  network, no snapshots. Useful for benchmarking the microkernel path,
  but not appropriate for production. The provider exposes it for
  completeness and refuses to register tools or apply a network
  allowlist on a Nanvix session.

## Out of scope (deferred to follow-up designs)

* Custom guest binaries beyond the upstream Python and JS packages.
* Multi-tenant pooling (one sandbox shared across agents). Today every
  session is its own VM; pooling is a separate trust analysis.
* Cross-backend feature parity. We accept that `nanvix` is partial; the
  ABC's `is_available()` should expose this.
* Telemetry hooks. Upstream does not yet emit structured per-call metrics.
  When it does, AGT should map them into the existing `agent-sre`
  observability surface.
